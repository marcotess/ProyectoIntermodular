# Estado del proyecto frente a la rubrica

## Objetivo de este documento

Este archivo resume que partes del Proyecto Fin de Ciclo ya tienen base en el repositorio, que partes estan incompletas y que entregables concretos faltan para acercarse a la maxima puntuacion de la rubrica.

## Valoracion general actual

El proyecto ya dispone de una base tecnica suficiente para defender bien:

- tecnologias utilizadas;
- arquitectura general;
- parte del analisis funcional;
- parte del modelo de datos;
- parte de la codificacion.

Las carencias principales no estan tanto en el codigo como en la evidencia documental exigida por la rubrica:

- memoria final maquetada;
- diagramas;
- manuales completos;
- viabilidad con cifras;
- plan de pruebas con ejemplos reales;
- presentacion para la defensa.

## Revision por apartados

### 0. Estructura y formato

Estado: incompleto.

Ya existe una base en Markdown, pero falta:

- portada final;
- indice paginado;
- revision ortografica final;
- formato Verdana 10, interlineado 1.5, margenes y numeracion;
- PDF final listo para entregar.

Accion recomendada:

- pasar la memoria a documento final en procesador de texto;
- revisar estilo, ortografia y numeracion antes de exportar.

### 1. Introduccion

Estado: aceptable, pero mejorable.

Ya tienes contexto del problema y perfiles funcionales, pero para aspirar a la nota maxima todavia conviene desarrollar:

- el tipo de entorno o sector al que responde la aplicacion;
- la necesidad concreta que cubre;
- la organizacion del escenario academico;
- el rol de cada perfil dentro del flujo documental.

Accion recomendada:

- ampliar con una descripcion mas narrativa del problema y de los actores implicados.

### 2. Tecnologias utilizadas

Estado: bastante bien encaminado.

Ya existe stack y justificacion tecnica, pero puede subir de nivel si anades comparativas breves frente a alternativas.

Ejemplo:

- Laravel frente a Symfony o Node.js para un PFC de alcance medio;
- Blade frente a un frontend desacoplado completo;
- Sanctum frente a otros mecanismos de autenticacion.

### 3. Arquitectura

Estado: bien encaminado.

La arquitectura en capas ya esta descrita. Para reforzarla, conviene anadir:

- un diagrama de arquitectura;
- ventajas del modelo elegido;
- relacion entre controladores, acciones, modelos y middleware.

### 4. Analisis de requisitos

Estado: medio.

Ya hay requisitos funcionales y no funcionales, pero faltan evidencias de captacion y formalizacion:

- casos de uso completos;
- historias de usuario o UML de casos de uso;
- fases del proyecto con temporalizacion;
- justificacion del ciclo de vida seguido.

### 5. Diseno estatico

Estado: pendiente.

Faltan diagramas entregables:

- diagrama de clases UML;
- diagrama de secuencia;
- diagrama de un caso de uso.

### 6. Modelo de datos

Estado: medio.

Ya tienes entidades y relaciones identificadas, pero falta el material evaluable:

- diagrama E-R;
- explicacion tabla por tabla;
- justificacion de relaciones principales.

### 7. Codificacion

Estado: razonablemente bien.

Ya hay base para defender:

- validaciones;
- control de acceso por roles;
- manejo de errores;
- estructura modular.

Todavia conviene mejorar la presentacion del apartado con:

- fragmentos de codigo comentados y explicados;
- una seccion especifica de accesibilidad y usabilidad;
- ejemplos concretos de try/catch, validacion y permisos.

### 8. Manuales

Estado: incompleto.

Existe una base escrita, pero faltan:

- manual de instalacion detallado;
- manual de usuario paso a paso;
- capturas reales;
- mapa de pantallas.

### 9. Viabilidad

Estado: flojo.

Este es uno de los apartados mas faciles de perder si no anades contenido cuantificable. Faltan:

- estimacion temporal;
- coste de desarrollo;
- coste de infraestructura;
- posibles fuentes de financiacion;
- riesgos laborales, medioambientales e informaticos.

### 10. Pruebas de software

Estado: flojo.

La documentacion ya propone pruebas, pero en el repositorio solo hay una prueba trivial de ejemplo. Para acercarte al maximo debes incorporar:

- un plan de pruebas formal;
- al menos 3 ejemplos desarrollados;
- si es posible, pruebas automatizadas reales.

### 11. Conclusiones y mejoras futuras

Estado: base correcta, pero todavia corta.

Falta comparar claramente:

- plan inicial;
- resultado conseguido;
- limitaciones actuales;
- mejoras futuras realistas.

### 12. Bibliografia y webgrafia

Estado: aceptable.

Ya existe un listado base, pero conviene:

- citar fuentes dentro del documento final;
- mantener un estilo uniforme de referencias.

### Exposicion

Estado: pendiente.

No hay todavia material visible para la defensa. Faltaria preparar:

- presentacion visual;
- guion de 15 minutos;
- reparto de tiempos;
- respuestas a preguntas probables del tribunal.

## Prioridad recomendada

Si quieres subir nota rapido, este seria el orden correcto:

1. diagramas;
2. manuales con capturas;
3. pruebas de software;
4. viabilidad con cifras;
5. memoria final maquetada;
6. presentacion.

## Minimo para dejarlo competitivo

Para que el trabajo quede claramente por encima de una entrega media, deberias cerrar como minimo:

- memoria final completa en PDF;
- 4 diagramas reutilizables: casos de uso, clases, secuencia y E-R;
- 1 manual de instalacion;
- 1 manual de usuario con capturas;
- 3 pruebas bien explicadas;
- 1 seccion de viabilidad con numeros basicos;
- 1 presentacion de defensa.
